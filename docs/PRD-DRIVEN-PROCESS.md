## 🎯 核心目标

通过 PRD (Product Requirement Document) 驱动开发任务自动拆解和执行，实现：
- 需求 → 任务自动转化
- 减少人工干预
- 提高任务拆解准确性
- 保持上下文一致性

## 📁 PRD 文档结构

### 标准模板

```markdown
# PRD: {项目名称}

**版本**: 1.0  
**创建日期**: YYYY-MM-DD  
**负责人**: {负责人姓名}  
**优先级**: P0/P1/P2/P3

---

## 1. 项目概述

### 1.1 背景
{为什么需要这个项目}

### 1.2 目标
{项目要达成什么目标}

### 1.3 范围
{项目包含什么，不包含什么}

---

## 2. 功能需求

### 2.1 功能列表
| 编号 | 功能名称 | 描述 | 优先级 |
|------|----------|------|--------|
| F001 | {功能名} | {描述} | P0/P1/P2 |

### 2.2 功能详情

#### F001: {功能名称}
- **用户故事**: 作为 {角色}，我希望 {功能}，以便 {价值}
- **验收标准**:
  - [ ] 标准 1
  - [ ] 标准 2
- **技术要点**:
  - 技术栈：{技术}
  - 接口：{API 设计}
  - 数据：{数据模型}

---

## 3. 非功能需求

### 3.1 性能要求
- 响应时间：< 500ms
- 并发用户：> 100

### 3.2 安全要求
- 认证：JWT
- 授权：RBAC

### 3.3 兼容性要求
- 浏览器：Chrome, Firefox, Safari
- 系统：macOS, Windows, Linux

---

## 4. 技术架构

### 4.1 系统架构图
{架构图或描述}

### 4.2 技术栈
- 前端：Vue 3 + Element Plus
- 后端：FastAPI + SQLite
- 部署：Docker

### 4.3 数据模型
{数据库设计}

---

## 5. 项目计划

|------|------|--------|
| Phase 1 | 第 1-2 周 | 基础功能 |

### 5.2 依赖关系
- 依赖项目：{项目名}

- [ ] 性能测试通过
```

### 工作流程
1. **接收 PRD** - 从擎天柱接收已初审的 PRD
2. **技术拆解** - 分析 PRD 生成 Task Spec 和 Dev Spec
3. **任务分配** - 根据任务类型分配给忍者神龟成员
4. **架构审查** - 审查代码实现是否符合架构设计
5. **进度汇报** - 每小时汇报开发进度
